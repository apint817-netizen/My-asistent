# Руководство по решению проблем с ИИ (AI Troubleshooting)

Этот документ описывает проблему с интеграцией Google Gemini, с которой мы столкнулись на этапе деплоя, и её успешное решение. Если ИИ перестанет работать или появятся ошибки 400/404/429, обращайтесь к этому файлу.

## 🔴 Симптом: ИИ выдает 400 или 404 ошибку (no body)

**Описание бага:** 
После деплоя на Vercel ИИ начинал безостановочно отдавать ошибки вида:
- `Ошибка при обращении к ИИ: 400 status code (no body)`
- `Ошибка при обращении к ИИ: 404 status code (no body). Проверьте правильность настроек API`
- Возможны частые падения по квоте (429 Too Many Requests), даже если запросов мало.

**Причина:**
Это происходило из-за использования официального npm-пакета `openai` (SDK) в комбинации с базовым URL от Google (`https://generativelanguage.googleapis.com/v1beta/openai`). 

Хотя Google анонсировал поддержку OpenAI SDK, этот "мост" в их инфраструктуре **ещё работает нестабильно для сложных чатов**. Когда мы отправляем длинную историю сообщений, системный промпт или запрашиваем новейшие модели (например, `gemini-2.0-flash`), OpenAI SDK формирует URL-запрос, который серверы Google не могут распознать или правильно распарсить, отвечая пустыми HTTP-ошибками (400 или 404). 

Также часть ошибок 429 была связана с тем, что на клиенте "залипал" старый демо-ключ, который перекрывал серверный ключ.

## ✅ Решение: Двойной подход (Dual Approach)

Чтобы обеспечить 100% стабильность, мы реализовали гибридную архитектуру, вдохновленную проектом `finmodel-ai`.

### 1. Нативный Google REST API для чата (`api/chat.js`)
Главный обработчик чатов больше **не использует** пакет `openai`. Слишком сложные запросы и системные промпты мы формируем вручную и отправляем напрямую `fetch`-запросом в родной API гугла.

```javascript
// Пример правильного вызова в api/chat.js:
const response = await fetch(`https://generativelanguage.googleapis.com/v1beta/models/${modelName}:generateContent?key=${API_KEY}`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(body)
});
```
Этот подход **абсолютно надежен**, он прозрачно отдает причины ошибок (если они есть) и безупречно работает со всеми новыми моделями. В нем также зашит **3-уровневый Fallback**:
Если `gemini-2.0-flash` недоступна, сервер автоматически переберет `gemini-1.5-flash`, `gemini-1.5-pro` и `gemini-pro`.

### 2. OpenAI SDK для простых задач (`api/validate.js`)
Там, где нам нужен структурированный ответ в формате JSON (например, при проверке названия задачи на осмысленность), мы оставили пакет `openai` в связке с кастомным клиентом.

```javascript
// api/_lib/ai-client.js
export function getAIClient(clientApiKey = null) {
    const apiKey = clientApiKey || process.env.GOOGLE_API_KEY;
    return new OpenAI({
        apiKey: apiKey,
        baseURL: 'https://generativelanguage.googleapis.com/v1beta/openai',
    });
}
```
Он используется в `/api/validate.js` с параметром `response_format: { type: "json_object" }`. Для коротких системных задач без сложной истории чата этот мост работает отлично.

### 3. Приоритет API ключей
Мы внедрили жесткую логику изоляции:
1. Ключ пользователя, введенный в настройках UI, 100% передается в `Authorization` header.
2. Безопасность: на сервере в `api/chat.js` и `api/validate.js` заблокирован битый "дефолтный" ключ (`AIzaSy...`), чтобы он не перекрывал серверный `process.env.GOOGLE_API_KEY`.

Таким образом, ИИ теперь всегда работает либо на личном ключе юзера (избегая 429 для сервера), либо падает на надежный бэкенд Google REST API.
